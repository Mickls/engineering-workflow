# 逻辑原型

用于回答“这个状态机、规则、数据模型或 API 形状是否合理”。

除非用户或项目规则明确要求其他语言，逻辑原型说明、结论记录和交付摘要默认使用中文；文件名、命令、API、代码标识符、状态枚举和引用路径可以保留英文。

## 形态

- 轻量命令行或脚本。
- 状态放内存。
- 使用项目已有语言和运行方式。
- 核心逻辑放在可移植的小模块中，TUI/CLI 只是壳。
- 默认写入 `.codex/engineering-workflow/prototypes/`；如果绑定具体需求，写入对应 `.codex/engineering-workflow/issues/<REQ-or-EPIC>/`。
- 提供一条明确运行命令。

## 过程

1. 写清问题：原型到底要验证什么。
2. 建一个最小状态模型或输入输出模型。
3. 提供少量用户可触发动作。
4. 每次动作后打印完整关键状态。
5. 让用户或 agent 跑典型和边界场景。
6. 记录结论：保留哪条规则，推翻哪条假设。

## 反模式

- 为原型写完整测试。
- 接真实生产数据库。
- 为未来可能需求泛化。
- 把 terminal shell 代码直接带进生产实现。
- 原型没有回答问题却继续打磨。
