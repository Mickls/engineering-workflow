# 需求工作流

## 概述

先理解需求和现有实现，再决定怎么改。这个 skill 负责把用户的口语化需求转成可执行、可审查、可追踪的中文 `issue/design/plan` 文档。

优先读取项目上下文：

- `docs/agents/project-profile.md`
- `docs/agents/issue-workflow.md`
- `docs/agents/domain.md`
- `CONTEXT.md` / `CONTEXT-MAP.md`
- 相关 `docs/adr/`

如果这些文档缺失且会影响多个后续任务，建议先使用 `project-setup`。

可按需读取参考模板：

- [design-template.md](references/design-template.md)
- [plan-template.md](references/plan-template.md)
- [coverage-matrix.md](references/coverage-matrix.md)
- [clarification-checklist.md](references/clarification-checklist.md)

## 使用场景

以下任务应使用本 skill：

- 新功能、bugfix、重构、接口变更、数据结构变更、流程变更。
- 用户描述还不够明确，需要对齐术语、边界、验收标准。
- 需要创建或更新 `issue/` 下的需求文档。
- 用户中途修改需求，可能影响已确认方案。
- 需求跨模块、跨多天或需要拆分子需求。

以下任务可以跳过完整 issue 文档：

- 纯解释、代码 review、格式化、注释、简单文案、无行为变化的小整理。
- 用户明确要求“直接修改、不写文档”。

跳过文档不等于盲改。仍需阅读相关代码，并说明判断为轻量任务的原因、修改范围和验证方式。

## 复杂度判定表

选择足够清晰但不过度复杂的流程：

| 类型 | 适用场景 | 文档要求 |
| --- | --- | --- |
| 轻量需求 | 纯解释、只读 review、格式/注释/文案、无行为变化的小整理、用户明确“直接改/不写文档” | 不创建 issue；最终说明轻量判断、范围、验证 |
| 普通需求 | 单个功能或 bugfix、单条主调用链、影响范围清晰 | 创建一个 `REQ-*` 目录，包含 `README.md`、`design.md`、`plan.md` |
| 大需求 / 跨模块需求 | 多阶段、多模块、多里程碑、存在明显依赖或风险拆分 | 创建 `EPIC-*` 目录和子需求目录 |

不确定时先按普通需求处理。如果 plan 已经膨胀到无法用一条清晰实施路径表达，应在编码前升级为 EPIC。

## 需求范围基线

除非用户明确要求 `MVP`、最小实现、临时方案、只做第一阶段或先交付可用雏形，否则 design/plan 必须按完整功能设计。

完整功能设计要求：

- 覆盖用户描述中可合理推导出的完整业务闭环，而不是只覆盖最容易编码的 happy path。
- 同时考虑主流程、异常流程、权限/鉴权、数据一致性、兼容性、配置、日志、可观测性、迁移、回滚、测试和交付验证。
- 对接口、数据表、proto/API、消息、任务、缓存、外部依赖等相关边界给出完整设计，不能只写“后续补充”。
- 如果完整功能过大，应拆成阶段、子需求或 EPIC，但文档必须保留完整目标、阶段边界、依赖关系和最终验收标准。
- 如果 plan 已经大到难以独立实现或验证，使用 `issue-slicing` 将其拆成可完成的垂直切片。
- 阶段拆分是为了控制执行顺序，不是默认缩减需求范围；每个被后置的能力都必须说明原因、触发条件和回到计划中的方式。
- `非目标范围` 只用于记录用户明确不需要、本轮确实无关、或因风险/依赖必须后置的内容，不得把完整功能中的必要能力随意放入非目标。
- 如果成本、风险或信息缺口导致无法一次设计完整功能，应明确列为开放问题或风险，请用户确认，而不是自动降级为 MVP。

当用户明确要求 MVP 时：

- 在 `design.md` 中写清 MVP 范围、完整功能范围和本次暂不做的内容。
- 在 `plan.md` 中写清从 MVP 演进到完整功能的后续步骤和补齐条件。
- 不允许把 MVP 描述成最终完整方案，除非用户明确确认完整需求只到该范围为止。

## 需求分类

### 修改型需求

已有相关实现，需要变更行为、字段、接口、流程或返回值。

必须调查：

- 找到入口：`HTTP/RPC handler`、`CLI`、`event/job`、定时任务、消息消费等。
- 追溯完整调用链：调用方、被调用方、数据来源、返回值、错误处理、副作用、已有测试。
- 理解当前实现目的：为什么已有逻辑这样写，依赖哪些业务假设。
- 对齐差异：用户期望和当前行为具体差在哪里。
- 标记不明确点：会影响实现路径的问题必须先问用户；能通过代码确认的不要问。

### 新增型需求

需要新增功能、接口、数据结构、任务、页面或模块。

必须调查：

- 查阅相邻模块，理解项目中相关术语的真实含义。
- 找到相似实现：接口、DAO、service、job、测试、配置。
- 确认新增内容如何接入现有边界：入口、权限、存储、错误、日志、测试。
- 不清楚但不阻塞实现的点，写为显式假设。

## Issue 结构

默认在 workspace 根目录维护 `issue/`。

## Issue 维护流程

创建或更新 issue 文档时：

1. 先检查现有 `issue/` 目录，再决定新编号。
2. 普通需求使用 `REQ-YYYYMMDD-NNN`，其中 `NNN` 是当天未使用的下一个序号。
3. 大需求使用 `EPIC-YYYYMMDD-NNN`，序号规则相同。
4. 需求简述保持短、可读、稳定；不要因为措辞变化随意重命名目录。
5. 当存在多个需求目录时，创建或更新 `issue/README.md`。
6. 新需求写作时状态为 `draft`，准备给用户确认时改为 `reviewing`。
7. 只有用户确认 design/plan，或明确要求继续实现时，状态才改为 `approved`。
8. 开始实现时状态改为 `doing`。
9. 只有存在验证证据后，状态才改为 `done`。
10. 如果被阻塞，状态改为 `blocked`，并在 `README.md` 或 `risks.md` 记录阻塞原因。

不要只根据创建时间判断执行顺序，应使用 `issue/README.md`、`roadmap.md` 或显式依赖。

### 轻量需求

不强制创建 issue 文档。最终回复必须说明：

- 为什么是轻量需求。
- 修改范围。
- 验证方式。

### 普通需求

适用于单个功能、单条调用链、影响范围清晰的变更：

```text
issue/REQ-YYYYMMDD-NNN_需求简述/
  README.md
  design.md
  plan.md
```

规则：

- `REQ-YYYYMMDD-NNN` 是稳定需求编号，`NNN` 为当天递增序号。
- `需求简述` 保持短、可读、可搜索。
- `README.md` 放状态、优先级、依赖、阻塞项和当前结论。
- `design.md` 描述需求和方案。
- `plan.md` 描述实施步骤和验证方式。

### 大需求 / 跨模块需求

适用于多个阶段、多个功能包、多个里程碑：

```text
issue/EPIC-YYYYMMDD-NNN_主题简述/
  README.md
  roadmap.md
  risks.md
  001_子需求A/
    README.md
    design.md
    plan.md
  002_子需求B/
    README.md
    design.md
    plan.md
```

规则：

- `README.md` 是总入口：目标、范围、状态总表、阅读顺序。
- `roadmap.md` 说明阶段、优先级、依赖关系和交付顺序。
- `risks.md` 统一登记风险、门禁和后置项。
- 子需求目录按执行顺序编号。
- 每个子需求都有独立 `design.md` 和 `plan.md`。

### 跨日期持续需求

同一个需求跨多天推进时，继续维护原需求目录，不因日期变化复制一份新的 design/plan。

在 `README.md` 或 `plan.md` 追加：

- 当前状态。
- 最近更新时间。
- 已完成事项。
- 新增决策。
- 剩余任务。

### issue 总入口

当 `issue/` 下存在多个需求目录时，维护：

```text
issue/README.md
```

至少包含当前推荐执行顺序、需求状态总表、优先级总表、依赖关系、阻塞项、已完成和后置需求。

## README 元信息

普通需求和子需求的 `README.md` 顶部必须包含：

```md
---
id: REQ-YYYYMMDD-NNN
title: 需求标题
status: draft | reviewing | approved | doing | done | blocked | deferred
priority: P0 | P1 | P2
type: feature | fix | refactor | docs | chore
depends_on:
  - REQ-xxx
blocks:
  - REQ-yyy
updated_at: YYYY-MM-DD
---
```

状态含义：

- `draft`：草稿，尚未自审完成。
- `reviewing`：等待用户确认。
- `approved`：已确认，可进入实现。
- `doing`：正在实现。
- `done`：已完成并验证。
- `blocked`：被开放问题、依赖或风险阻塞。
- `deferred`：明确后置。

## design 文档

`design.md` 说明“需求是什么、为什么做、准备怎么做”，至少包含：

- 需求背景：问题来源、当前行为、目标用户或业务场景。
- 需求总结：本次要改变什么，不改变什么，完成后的可观察结果。
- 详细需求：完整业务闭环、功能规则、数据流、接口定义、数据表结构、proto/API 变更、错误处理、兼容性、权限、边界条件、相关文档或引用。
- 关键约束覆盖表：把必须实现、不能发生、边界条件、兼容性要求、数据安全要求、验收标准提炼成可追踪条目。
- 非目标范围：本次明确不做什么。
- 验收标准：用户或测试如何判断需求完成。
- 关键假设：尚未完全确认但不阻塞推进的前提。
- 兼容性策略：对旧数据、旧接口、旧调用方、已有配置的影响。
- 完整性检查：说明是否覆盖完整功能；如果存在阶段拆分或后置能力，说明原因和恢复路径。

如果涉及数据库、接口、消息、配置或外部依赖，必须讨论清楚，不能只写概念。

### 关键约束覆盖表

非轻量需求必须在 `design.md` 中维护关键约束覆盖表。不要只把关键边界散落在自然语言段落里。

关键约束包括：

- 用户明确说“必须”“不能”“应当”的行为。
- 会导致数据丢失、权限绕过、重复写入、状态错乱、兼容性破坏的边界。
- 自动流程和手动流程语义不同的地方。
- existing / non-existing、empty / invalid、root 内外、include / exclude、sync / async、old / new config 等容易被实现遗漏的分支。
- `验收标准` 中每一条可观察行为。

覆盖表格式：

```md
| ID | 类型 | 关键约束 | 影响入口/调用链 | 实现要求 | 验证意图 |
| --- | --- | --- | --- | --- | --- |
| C-001 | boundary | ... | ... | ... | ... |
```

规则：

- ID 在同一需求内稳定，例如 `C-001`、`C-002`。
- 一条约束只表达一个必须被实现或验证的事实；不要把多个独立边界塞进同一行。
- 如果一个自然语言段落包含多个“必须/不能/即使/除非”，应拆成多条约束。
- 如果某条约束无法自动测试，仍然保留 ID，并在验证意图中写清手动验证或代码审查证据。
- 不允许在 plan 或实现阶段静默忽略约束；如果发现约束不可行，先更新 design/plan 并等待用户确认。

## UI 原型门禁

当需求同时满足以下条件时，必须在用户确认前生成静态 HTML 原型：

- 需求是独立需求：可以作为一个完整页面、模块、弹窗、表单、流程或工具独立理解和确认，而不是仅修改少量文案、颜色、间距或已有页面中的局部样式。
- 需求包含 UI 相关改动：新增或调整页面布局、组件结构、交互流程、表单、表格、图表、列表、导航、弹窗、状态展示、空态、错误态或成功态。

如果需求不是 UI，但涉及复杂状态机、业务规则、数据模型或 API 形状，考虑使用 `prototyping` 的 logic prototype 先验证方案。

原型要求：

- 原型默认放在对应 issue 目录下；单页面使用 `prototype.html`，多页面或需要资源时使用 `prototype/index.html`。
- 原型必须是可直接打开的静态 HTML；除非项目已有明确原型规范，不引入构建流程，不写入生产代码目录。
- 原型应尽量贴近项目现有 UI 风格。生成前需要查看相邻页面、组件、样式或设计系统；如果没有可参考内容，再采用朴素但完整的静态样式。
- 原型需要覆盖完整功能的主要 UI 闭环，至少展示默认态、关键操作态、空态、错误态或成功态中与需求确认相关的状态。
- 可以使用 mock 数据和前端内联交互来表达效果，但必须在 `design.md` 中说明这些内容不是最终数据或生产实现。
- 如果静态 HTML 无法准确表达关键交互，需要在 `design.md` 中写明限制，并补充截图说明、状态列表或交互说明；除非用户明确跳过，不得因此省略原型确认。
- 原型不得替代 `design.md` / `plan.md`。它只是用于确认视觉结构、交互意图和页面状态。

原型自检和迭代边界：

- 原型是需求确认草稿，不是生产前端实现，不进入编码交叉 review、生产 lint/test/build 或生产级验收循环。
- 生成原型后只做一次有限自检：确认文件可打开、核心页面或状态可见、关键 mock 交互可触发、没有阻塞用户理解的明显布局错乱。
- 如果发现阻塞用户理解的问题，可以做最多一轮修正并重新自检；修正后仍存在的问题应记录为原型限制或待确认问题，交给用户 review。
- 不得围绕原型反复执行“review -> 修改 -> 再 review -> 再修改”的循环；除非用户明确要求继续打磨，否则原型达到可表达设计意图后必须停止。
- 非阻塞的视觉精修、动效、像素级对齐、真实数据接入和生产组件复用，不应在原型阶段追求完成；需要时写入 `design.md` 或 `plan.md` 的待确认项或实现阶段任务。
- 如果用户明确要求审查原型，审查范围仅限是否能表达需求和关键状态，不扩展为生产代码审查。

文档联动要求：

- `design.md` 必须记录原型路径、覆盖的界面状态、未覆盖的交互和需要用户确认的 UI 问题。
- `plan.md` 必须包含“UI 原型确认”步骤；在用户确认设计、计划和原型前，不得进入生产编码。
- 如果需求被拆成多个子需求，只有包含 UI 改动且可独立确认的子需求需要生成对应原型。

## plan 文档

`plan.md` 说明“具体做什么、按什么顺序做、每步完成后达到什么目的”，至少包含：

- 总目标和验收标准。
- 任务拆分和执行顺序。
- 每个任务的修改范围、依赖关系、产出、验证方式。
- 涉及的文件或模块。
- 需求覆盖矩阵：每个关键约束 ID 对应的实施步骤、代码落点、测试或手动验证方式。
- 风险点和需要用户确认的开放问题。
- 完整功能交付路径：如果分阶段执行，说明每个阶段如何共同达成完整功能，而不是只交付 MVP 后结束。

复杂需求继续拆分阶段，例如：数据库结构、DAO 通用逻辑、接口/proto、业务逻辑、调用方接入、测试和迁移。

### 需求覆盖矩阵

非轻量需求的 `plan.md` 必须把 `design.md` 的关键约束 ID 映射到实施和验证。

推荐格式：

```md
| 约束 ID | 实施步骤 | 预计代码落点 | 测试/验证入口 | 状态 |
| --- | --- | --- | --- | --- |
| C-001 | Step 2 | path/to/file.go:function | integration test / manual command | planned |
```

规则：

- `design.md` 中每个关键约束 ID 都必须出现在覆盖矩阵里。
- 一个测试可以覆盖多个约束；不要为了表格一行一测而制造低价值测试。
- 如果某个约束只适合代码审查或手动验证，必须写清原因、验证命令或检查证据。
- 如果一个实施步骤没有关联任何约束，检查它是否属于无关扩展；除非它是必要的重构、迁移或支撑任务，否则不要做。
- 编码阶段需要回填实际代码落点、测试名、验证命令和状态；最终交付前不得留下未解释的 `planned` / `todo`。

## 文档自审门禁

生成 design/plan 后必须自审：

- 是否存在互相矛盾的需求或步骤。
- 是否有无法落地的笼统描述。
- 是否遗漏数据结构、接口、错误处理、兼容性或测试策略。
- 是否拆分过大，导致单个任务难以安全执行。
- 是否存在需要用户确认但未标记的问题。
- 是否在用户未要求 MVP 的情况下，把必要能力降级为 MVP、后置项或非目标范围。
- 分阶段计划是否仍然能达成完整功能，且每个后置项都有原因、触发条件和恢复路径。
- design 中关键约束是否已编号，plan 中是否逐条映射到实施和验证。
- 是否存在关键约束只写在段落里，没有进入覆盖表和覆盖矩阵。
- 独立 UI 需求是否已生成静态 HTML 原型，并在 design/plan 中记录原型路径、覆盖状态和待确认问题。
- UI 原型是否只做了有限自检；如有问题，是否避免了重复 review/修正循环，并把非阻塞问题交给用户确认。

自审完成后，如果创建或更新了 `design.md` / `plan.md`，必须进入用户确认门禁。

## 用户确认门禁

这是硬门禁，不是建议：

- 创建或更新 `design.md` / `plan.md` 后，必须停止当前回合，请用户 review 和确认。
- 如果维护了 issue 状态，此时应保持或更新为 `reviewing`，不得提前改为 `approved` 或 `doing`。
- 如果本需求触发 UI 原型门禁，必须把原型和 design/plan 一起交给用户确认。
- 在后续用户消息明确确认前，不得进入实现阶段。
- 同一轮中，即使 agent 自己认为方案已经完整、没有开放问题，也不得继续编码。
- 不得执行任何实现类动作，包括修改业务代码、生成 migration/proto、编写或调整测试、运行实现任务，或安排任何实现类/代码审查类工作。
- 允许做的收尾动作仅限于文档自审、原型有限自检、回读检查、轻量格式校验，以及在回复中列出文档路径、原型路径、摘要、开放问题和需要用户确认的点。

只有用户明确要求跳过文档确认、或明确要求“写完方案后直接实现”，才可以越过本门禁；这类例外必须在回复中说明。

除非用户明确要求跳过文档确认，以下确认必须来自创建或更新文档之后的后续用户消息。以下情况视为用户确认：

- 用户明确说“按这个方案做”、“开始实现”、“可以”、“继续”。
- 用户明确要求“直接修改”或“不写文档”，且任务已被判断为轻量，或用户接受跳过文档。
- 用户在已有 design/plan 上只提出非阻塞修改，并明确允许同步更新后继续。

以下情况不算用户确认：

- agent 自己认为方案合理。
- 用户只是在补充背景，没有表示可以实现。
- design/plan 中仍有会影响实现路径的开放问题。

如果缺少确认，应在写完或更新文档后停下，请用户 review。

## 需求变更

- 用户变更需求后，先判断是否影响已确认的 design/plan。
- 影响实现路径、接口、数据结构、验收标准或测试策略时，先更新文档，再继续编码。
- 不允许在代码实现中偷偷偏离已确认的 design/plan。
- 如果已确认方案在实现层面不可行，暂停并说明原因、影响范围和替代方案。

## 外部资料

- 第三方库、框架、SDK、OpenAPI、数据库特性或可能随版本变化的 API，优先使用 Context7 或官方文档确认。
- 项目内部业务逻辑、调用链路和约定以仓库代码、测试和内部文档为准。
- 不凭记忆使用可能已变化的 API、配置项或命令。
- 引用外部资料时，在 `design.md` 中记录关键链接、版本或结论。
- 如果 Context7 或网络不可用，记录不可用原因，并优先使用项目 lockfile、vendor、README、源码注释和本地文档作为兜底依据。
