# UI 原型门禁

当需求同时满足以下条件时，必须在用户确认前生成静态 HTML 原型：

- 需求是独立需求：可以作为完整页面、模块、弹窗、表单、流程或工具独立理解和确认。
- 需求包含 UI 相关改动：新增或调整布局、组件结构、交互流程、表单、表格、图表、列表、导航、弹窗、状态展示、空态、错误态或成功态。

## 原型要求

- 默认放在对应 `.codex/engineering-workflow/issues/<REQ-or-EPIC>/` 目录下；单页面使用 `prototype.html`，多页面或需要资源时使用 `prototype/index.html`。
- 必须是可直接打开的静态 HTML；不引入构建流程，不写入生产代码目录或生产路由。
- 尽量贴近项目现有 UI 风格。生成前查看相邻页面、组件、样式或设计系统。
- 覆盖完整功能的主要 UI 闭环，至少展示默认态、关键操作态、空态、错误态或成功态中与需求确认相关的状态。
- 可以使用 mock 数据和内联交互，但必须在 `design.md` 中说明这些不是最终数据或生产实现。
- 原型不得替代 `design.md` / `plan.md`。

## 自检和迭代边界

- 生成原型后只做一次有限自检：确认文件可打开、核心页面或状态可见、关键 mock 交互可触发、没有阻塞用户理解的明显布局错乱。
- 发现阻塞用户理解的问题，可以做最多一轮修正并重新自检。
- 不围绕原型反复执行 review / 修改循环；非阻塞视觉精修、动效、像素级对齐、真实数据接入和生产组件复用，应写入待确认项或实现阶段任务。

`design.md` 必须记录原型路径、覆盖状态、未覆盖交互和需要用户确认的 UI 问题。`plan.md` 必须包含“UI 原型确认”步骤。
