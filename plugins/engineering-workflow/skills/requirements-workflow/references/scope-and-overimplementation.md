# 范围和过度实现

## 完整功能基线

除非用户明确要求 `MVP`、最小实现、临时方案、只做第一阶段或先交付可用雏形，否则 design/plan 必须按完整功能设计。

完整功能设计要求：

- 覆盖用户描述中可合理推导出的完整业务闭环，而不是只覆盖 happy path。
- 同时考虑主流程、异常流程、权限/鉴权、数据一致性、兼容性、配置、日志、可观测性、迁移、回滚、测试和交付验证。
- 对接口、数据表、proto/API、消息、任务、缓存、外部依赖等边界给出完整设计，不能只写“后续补充”。
- 如果完整功能过大，应拆成阶段、子需求或 EPIC，但保留完整目标、阶段边界、依赖关系和最终验收标准。
- 阶段拆分是为了控制执行顺序，不是默认缩减需求范围。

当用户明确要求 MVP 时：

- `design.md` 写清 MVP 范围、完整功能范围和本次暂不做的内容。
- `plan.md` 写清从 MVP 演进到完整功能的后续步骤和补齐条件。
- 不把 MVP 描述成最终完整方案，除非用户明确确认完整需求只到该范围为止。

## 过度实现分类

新增能力按以下分类记录：

- `required`：当前用户目标、完整功能闭环、风险控制或验收标准必需。
- `defer-with-trigger`：当前不必实现，但未来触发条件明确；写清触发条件、恢复路径和为什么现在不做仍不破坏完整目标。
- `out-of-scope`：用户明确不需要、本轮无关、已有能力覆盖，或缺少业务证据的 speculative scope。

处理规则：

- 用户未明确要求且没有当前业务证据的 speculative feature，不得进入关键约束覆盖表。
- 仅为了未来可能变化而新增的配置项、抽象、单实现 interface、浅层 wrapper、备用依赖或目录结构，应默认归入 `defer-with-trigger` 或 `out-of-scope`。
- 标准库、框架、平台 native feature、项目已有工具或已安装依赖已经满足需求时，design/plan 记录该选择，不再设计自定义实现。
- 完整功能确实需要后置项时，不能用 YAGNI 隐藏；必须写入 `defer-with-trigger`。
