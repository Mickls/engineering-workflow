# 澄清访谈模式

目标是在 shared understanding 之前逐步解决决策，而不是一次抛出一组问题。

## 原则

- 事实先查证：能通过代码、测试、配置、README、现有 issue、项目上下文或相邻实现确认的事实，不问用户。
- 决策问用户：影响产品语义、实现路径、验收标准、兼容性、风险取舍、UI/交互、数据保留、权限、安全或交付范围的选择，必须交给用户确认。
- 一次只问一个问题；上一题得到答复或用户明确跳过后，再问下一题。
- 每个问题都给出推荐答案，并简短说明推荐理由、替代选项和取舍影响。
- 先沿依赖顺序问；上游决策会影响后续问题时，先解决上游。
- 如果答案不会改变实现路径或验收标准，记录为假设或风险，不打断用户。
- 未达成 shared understanding 前，不创建最终 design/plan，不进入编码，不安排实现类任务。

## 问题格式

```md
我建议选择 A，因为 ...。是否按 A 处理？
如果选 B，影响是 ...。
```

用户明确要求快速推进或跳过追问时，仍需保留最小关键约束、显式假设和风险。如果缺少某个决策会导致实现不可安全推进，必须停下问这一题。
