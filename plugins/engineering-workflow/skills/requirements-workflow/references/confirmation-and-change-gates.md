# 用户确认和需求变更门禁

## 澄清批准

AI 输出完整澄清核销总结后必须停止。只有用户在后续消息中明确表达“澄清完成”“全部确认”“按总结开始设计”等同等语义，才可创建或更新 design/plan。

回答单个问题的“可以”、补充背景、认可局部方向、表示“明白了”或含糊的“继续”不算整体批准。用户表示疲劳、暂停、看不动或稍后继续时，保持当前问题状态并停止新增问题。批准口径不清时必须说明缺少哪项确认并等待，禁止推断。

用户明知 unresolved unknown 和风险仍要求继续时，可以记录 informed override；记录必须列出被接受的未知、风险和授权消息，之后按用户决定设计。

## Design / Plan 批准

澄清批准后创建或更新 design/plan，状态改为 `reviewing` 并停止当前回合。review 只检查文档是否忠实表达已确认内容、约束是否完整映射、方案是否清晰可执行；不得把基础需求发现和重大技术决定留到该阶段。

设计批准前不得编码、写或调整测试、生成 migration/proto，或安排实现类任务。只有 design/plan 之后的后续用户消息明确表示“按方案实现”“开始实现”等同等语义，才改为 `approved` / `doing`。

不算 design 批准：agent 自认为合理、用户只补充背景、局部认可、或文档仍存在影响实现路径的未知。

## 需求变更和重新澄清

- 新信息影响范围、行为、接口、数据结构、事务、兼容、风险、验收或测试策略时，立即停止当前阶段并重新打开 `clarifying`。
- 沿决策依赖标记 `still-valid`、`needs-reconfirmation`、`invalidated`，只重问受影响分支，再生成新的核销总结。
- 澄清期间不把候选内容写进旧权威 design/plan；旧方案明确被否决时标记 `deprecated` / `superseded`。
- 已批准方案在实现层面不可行时，暂停并说明证据、影响和替代方向，不在代码中偷偷偏离。
