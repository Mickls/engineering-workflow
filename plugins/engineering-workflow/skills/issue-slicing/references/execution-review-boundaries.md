# AFK / HITL 执行和 Review 边界

本文件是切片连续执行、阻塞停点和 review budget 的唯一 owner。目标是让 agent 在已批准边界内连续完成可验证切片，同时把真正需要用户判断的变化升级为 HITL，而不是每完成一个阶段都固定等待。

## AFK 准入

design/plan 批准后默认优先按 `AFK` 连续执行；`HITL` 是证据触发的例外，不因模块多、并发、数据库或高风险标签本身自动逐切片等待。

切片只有同时满足以下条件才是 `AFK`：

- 行为、验收场景和约束 ID 已批准。
- 切片经过用户可观察入口或稳定 public contract，完成后可独立验证。
- 没有新增产品决定、风险取舍或会改变 owner / 长期边界的技术方向。
- 项目契约或相邻实现足以决定命名、目录、错误、依赖和测试风格。
- 存在真实入口或稳定 public contract 的验证方式。
- 未命中用户在 plan 中设置的 review budget。
- 预计实现不偏离 design/plan，也不扩大已批准文件或模块边界。

AFK 切片完成后，agent 对照约束、项目风格和验证证据自审；全部满足时直接继续下一个 AFK 切片。阶段性汇报可以发送，但不等待确认。

已批准设计覆盖的 API、数据、安全、并发等高风险内容不因标签本身自动逐切片阻塞；只有实现出现未覆盖风险、偏离或证据缺口时升级 HITL。最终交付仍按风险执行严格 review 和验证。

## HITL 触发

以下任一情况必须暂停：

- 出现新的用户决定，或已有决定需要重新确认。
- 正确实现需要偏离 design/plan、扩大范围或改变验收。
- 找不到适用项目先例，或候选实现与现有代码风格冲突。
- 无法从真实入口或稳定 public contract 证明目标行为。
- 正确路径需要外部权限、手动生成、迁移、部署或用户操作。
- 用户设置的累计 review budget 已达到，或下一切片预计会超过。
- 发现设计未覆盖的数据、安全、权限、并发、兼容、成本或不可逆风险。
- 测试失败后需要改变已确认的生产行为，而不是修正实现或测试。

HITL 必须说明实际场景、触发证据、受影响约束、为什么不能自行解决，以及用户只需判断的行为或风险。不得只用专业术语或完整 diff 代替问题说明。

## Review Budget

review budget 由用户或项目 plan 明确设置；未设置时写 `none`，不得自行套用固定行数。

plan 还必须声明 interaction budget：预计需要用户决定的问题数、预计阅读时间和阻塞停点。预算命中时先合并问题、应用已授权推荐项或暂停 interview 集中调查；不得把预算当成跳过 unresolved 高风险决定的理由。

如果使用代码行预算，plan 必须说明：

- 预算数值，例如 1500 行。
- 累计范围从哪个已 review 边界开始。
- 哪些文件属于核心 production source。
- additions / deletions 的统计方式。
- generated output、纯文档、fixture 和测试是否排除。

预算是限制 blast radius 的 hard gate，不证明实现正确，也不替代垂直切片、真实入口测试或风险判断。达到预算时停止在当前完整切片边界；下一切片预计会超过时，在开始前停止。

## Review Mode Reset

用户临时要求“每个单元 review”时，plan 必须记录原因、适用范围、截止条件和恢复 AFK 的条件，禁止把一次信任恢复措施继承到未命名的后续范围。达到 plan 中的截止条件后，连续两个切片没有方案偏离、未覆盖风险或验证缺口，或用户明确要求自行处理时，默认恢复 AFK；用户明确指定的 review 范围尚未结束时继续遵守。

手工 codegen、migration、wire、部署和大资源测试应集中到最少的自然里程碑。不要为了微型切片反复让用户手工接力；无法合并时说明真实依赖。

## Review 输出

AFK 连续执行期间只记录已完成切片、对应约束和验证证据。HITL 和最终交付使用 `verification-delivery` 的行为 / 风险 review 包，不要求用户先阅读完整 diff。
