# 诊断工作流

## 概述

普通 bug 也不能靠猜。先建立能稳定证明问题存在的反馈循环，再定位根因。线上事故、数据误删、幂等、并发等高风险问题优先使用 `incident-debugging`；本 skill 处理更一般的本地诊断。

## 阶段

### 1. 建立反馈循环

优先从真实入口或 public contract 的 failing test 开始；其次使用 HTTP/CLI/浏览器脚本、事件 replay 或最小 harness。非确定性和回归定位可使用 property/fuzz/stress、differential 或 `git bisect run` 循环；需要人工参与时参考 [hitl-loop.template.sh](scripts/hitl-loop.template.sh)。

反馈循环必须能观察用户描述的问题，而不是附近的另一个失败。

如果无法建立反馈循环，说明已尝试路径、缺失信息和下一步需要用户提供什么；不得直接猜测修复。

反馈循环质量检查：

- 速度：是否足够快，可以支持多轮假设验证。
- 确定性：是否稳定；如果是非确定性问题，是否已经通过循环、并发、固定种子、缩小时间窗等方式提高复现率。
- 断言精度：是否断言用户描述的具体症状、数据状态或副作用，而不是只断言“没有 crash”。
- 等价性：是否从真实入口或等价顶层入口触发问题；如果使用 harness，必须说明它和真实问题的等价点。

如果反馈循环慢、虚、或只覆盖附近失败，先改进反馈循环，不要进入修复。

### 2. 复现

- 运行反馈循环。
- 捕获失败输出、错误、数据状态或副作用。
- 对不稳定问题，提高复现概率，而不是假装稳定。

### 3. 提出假设

修改代码前列出 3-5 个可证伪假设。

```text
如果 X 是根因，那么改变/观察 Y 会让 Z 发生。
```

每个探针只能对应一个假设，避免一次改变多个变量。

没有可证伪假设和根因证据前，不得修改生产逻辑。可以添加带唯一前缀的临时 instrumentation，但必须在交付前清理。

### 4. Instrument

- 优先 debugger / REPL / targeted log。
- debug log 必须带唯一前缀，例如 `[DEBUG-a4f2]`。
- 不要“到处打日志再 grep”。
- 观测点必须服务具体假设，记录入口、关键输入摘要、状态变化、外部依赖结果或错误上下文中真正能缩小根因范围的信息。
- 如果现有日志、trace、metric 或状态证据不足以区分候选根因，先补充 targeted instrumentation 或向用户说明缺少什么信息；不得靠猜测修改生产逻辑。
- 性能问题先建立 baseline，再改代码。

### 5. 修复和回归

- 只改已证明根因。
- 如果存在正确测试 seam，先把最小复现固化成回归测试。
- 如果不存在正确 seam，记录这是架构或测试能力缺口。
- 不用兜底逻辑、吞错、重试、sleep、跳过步骤或默认值掩盖未证明根因。
- 修复后同时运行最小复现和原始反馈循环。
- 如果修复方案需要修改超出当前问题范围的生产逻辑，先说明影响范围并获得用户确认。

### 6. 清理和交付

- 删除所有 debug instrumentation，用唯一前缀 grep 确认。
- 删除或移动 throwaway harness。
- 汇报根因、验证命令、仍未覆盖风险。
- 如果只能确认信息不足，交付时说清已尝试路径、缺失观测点和建议新增的日志/trace/metric，不宣称已修复。

测试失败处理遵循 `testing-policy`，交付声明遵循 `verification-delivery`。
