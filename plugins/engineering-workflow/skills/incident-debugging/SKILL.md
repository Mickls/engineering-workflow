# 线上问题排查

## 目标

线上问题不能靠猜。必须先从真实入口建立复现证据，再证明根因，然后做最小修复，并用同一条复现路径证明问题已修复。

普通本地 bug 或失败测试优先使用 `diagnosis-workflow`。本 skill 用于线上事故、生产 bug、数据误删、重复写入、状态错乱、幂等、并发、重试、补偿、异步任务和消息消费等高风险问题。

## 硬门禁

以下门禁没有满足时，不得修改生产逻辑，也不得宣称已修复：

- 复现门禁：已经从真实入口或等价顶层入口复现问题。
- 根因门禁：已经用调用链和状态变化证明具体根因。
- 修复门禁：修改只针对已证明根因，不引入无关重构或防御式垃圾代码。
- 回归门禁：修复后重新运行同一条复现流程，并证明问题不再出现。
- 可观测性门禁：缺少日志、trace、metric、请求/响应、数据库状态或副作用证据时，只能补充观测点或汇报信息缺口。

## 事实收集

先整理事实，不要先猜原因：

- 用户现象、期望/实际行为、触发入口、步骤和重复次数。
- 必要的输入、租户/资源/时间范围，以及日志、请求、响应、数据库状态和副作用。

敏感信息只记录必要字段，不输出 secret、token、密码或完整隐私数据。临时 replay、harness、证据摘录或人工复现记录默认写入 `.codex/engineering-workflow/debug/` 或对应 issue 目录，不写入生产代码路径，默认不纳入 git。

## 复现门禁

复现必须尽量走真实入口：

- API/RPC 必须经过真实路由、参数绑定、鉴权/租户、handler 和持久化逻辑。
- 页面、event/job 和 CLI 使用对应真实入口或等价顶层输入。
- 数据误删、重复写入、状态错乱：检查真实持久化状态或等价测试数据库状态。

不允许用内部 helper、mock 掉问题本体、只断言中间函数返回值、不同简化路径或改写失败测试来伪造复现。

如果真实入口暂时无法复现，记录已尝试的命令、输入和结果，说明缺少什么信息或环境，并请求用户提供必要日志、数据或步骤；只能继续阅读代码和调查，不得基于猜测修改生产逻辑。

复现前应先建立反馈循环，例如 failing integration test、HTTP/CLI 脚本、事件 replay、最小 harness 或人机协作脚本。修改生产逻辑前，应提出 3-5 个可证伪假设，一次只验证一个假设。

临时 debug log 必须带唯一前缀并在交付前清理。长期保留的观测点应服务事故复盘和下次定位，优先记录 request/correlation id、tenant/user/resource id、job/message id、关键状态变化、外部依赖结果摘要和错误上下文。

## mock 和测试替身

允许 fake 或 mock 外部不可控依赖，例如第三方服务、支付网关、邮件、短信、远端 API。

不允许 mock 掉问题本体：核心调用链、数据库副作用、幂等检查、状态写入、权限或租户边界，除非已有上层测试覆盖且本次明确说明。

## 根因门禁

根因必须由证据支持，不能只写猜测。

需要追溯入口和请求解析、校验、权限、租户、幂等和并发控制、调用方和被调用方、数据读写删、事务和锁、外部副作用、错误处理和补偿逻辑。

根因说明至少包含：哪个具体入口触发问题、哪段调用链产生错误行为、哪个状态或数据变化证明问题、哪个代码位置是根因、为什么不是其他候选原因。

## 修复和回归

修复必须最小化：

- 只改已证明根因，不引入无关重构或无证据防御式判断。
- 不为测试改变需求语义，不删除或绕过关键业务逻辑。
- 不用临时日志、sleep、重试或兜底逻辑掩盖根因。
- 不吞错、不把未知异常转换成成功或默认结果；确需降级时必须有已确认业务契约、告警或日志，并能被回归验证观察到。

修复后必须重新运行同一条复现流程，使用修复前失败的入口和观察方式。不能用更低层、更短路径或只测 helper 的测试替代同路径回归。

## 汇报要求

最终汇报必须包含复现入口和命令、修复前失败证据、根因和关键代码位置、修改摘要、修复后同路径验证结果、仍未验证的范围和风险。

缺少复现证据、根因证据或同路径验证时，只能说“已调查到当前状态”，不能说“已修复”。如果当前结论是“信息不足”，必须列出已尝试的真实入口、缺失的数据或日志、建议新增的观测点，以及新增观测点后如何继续复现和定位。
