# 线上问题排查

## 概述

线上问题不能靠猜。必须先建立可复现证据，再定位根因，然后做最小修复，并用同一条复现路径证明问题已修复。

普通本地 bug 或失败测试优先使用 `diagnosis-workflow`。本 skill 用于风险更高的问题：线上事故、数据误删、幂等、并发、重复调用、状态错乱等。

本 skill 适用于：

- 线上问题、生产 bug、事故排查。
- 数据误删、重复写入、状态错乱、幂等性问题。
- 并发、重试、补偿、异步任务和消息消费问题。
- 用户明确要求“先复现再修”的问题。
- 已多次修复失败、现象反复出现、根因不清楚的问题。

## 硬门禁

以下门禁没有满足时，不得修改生产逻辑，也不得宣称已修复：

- 复现门禁：已经从真实入口或等价顶层入口复现问题。
- 根因门禁：已经用调用链和状态变化证明具体根因。
- 修复门禁：修改只针对已证明根因，不引入无关重构或防御式垃圾代码。
- 回归门禁：修复后重新运行同一条复现流程，并证明问题不再出现。

## 事实收集

排查开始时先整理事实，不要先猜原因：

- 用户看到的现象。
- 期望行为和实际行为。
- 触发入口：API/RPC、页面操作、event/job、CLI、消息、定时任务等。
- 触发步骤和重复次数。
- 关键输入、账号、租户、资源 ID、时间范围和环境。
- 相关日志、请求、响应、数据库状态和副作用。

敏感信息只记录必要字段，不输出 secret、token、密码或完整隐私数据。

临时 replay、harness、证据摘录或人工复现记录默认写入 `.codex/engineering-workflow/debug/` 或对应 `.codex/engineering-workflow/issues/<REQ-or-EPIC>/` 目录，不写入生产代码路径，默认不主动纳入 git。

## 复现门禁

复现必须尽量走真实入口：

- API/RPC 问题：调用同一个 API/RPC，经过真实路由、参数绑定、鉴权/租户处理、handler 和持久化逻辑。
- 页面问题：通过页面或等价的前端入口触发关键操作。
- event/job 问题：通过 event/job 接收入口或等价消息输入触发。
- CLI 问题：通过 CLI 命令入口触发。
- 数据误删/重复写入/状态错乱：必须检查真实持久化状态或等价测试数据库状态。

如果用户描述是“两次调用同样接口导致误删除”，复现必须至少包含：

- 构造能触发问题的真实或等价测试数据。
- 连续两次调用同一个接口入口。
- 比较第一次调用后、第二次调用后相关数据状态。
- 断言误删、重复删除、状态错乱或副作用确实出现。

不允许用以下方式伪造复现：

- 直接调用被怀疑的内部 helper 代替真实入口。
- mock 掉要验证的核心业务逻辑、持久化逻辑或幂等逻辑。
- 只断言某个中间函数返回值，不检查用户可见结果或真实副作用。
- 用与线上现象不同的简化路径证明问题存在。
- 复现失败后改写测试，让它看起来失败。

如果真实入口暂时无法复现：

- 记录已经尝试的复现命令、输入和结果。
- 说明缺少什么信息或环境。
- 请求用户提供必要日志、数据或操作步骤。
- 只能继续阅读代码和调查，不得基于猜测修改生产逻辑。

复现前应先建立反馈循环。可选形式包括 failing integration test、HTTP/CLI 脚本、事件 replay、最小 harness 或人机协作脚本。没有能证明问题的反馈循环时，不得进入修复。

修改生产逻辑前，应提出 3-5 个可证伪假设，并说明每个假设要通过什么观察或探针验证。一次只验证一个假设。

临时 debug log 必须带唯一前缀，例如 `[DEBUG-a4f2]`，交付前用该前缀 grep 确认已清理。

## mock 和测试替身

允许 fake 或 mock 外部不可控依赖，例如第三方服务、支付网关、邮件、短信、远端 API。

不允许 mock 掉问题本体：

- 要验证接口链路时，不 mock handler/service 的核心调用链。
- 要验证数据库副作用时，不 mock repository/DAO 到无法观察真实状态变化。
- 要验证幂等性时，不 mock 幂等检查或状态写入。
- 要验证权限或租户问题时，不绕过鉴权/租户边界，除非已有上层测试覆盖且本次明确说明。

## 根因门禁

根因必须由证据支持，不能只写猜测。

需要追溯：

- 入口和请求解析。
- 校验、权限、租户、幂等和并发控制。
- 调用方和被调用方。
- 数据读取、写入、删除、事务和锁。
- 外部副作用。
- 错误处理和补偿逻辑。

根因说明至少包含：

- 哪个具体入口触发问题。
- 哪段调用链产生错误行为。
- 哪个状态或数据变化证明了问题。
- 哪个代码位置是根因。
- 为什么不是其他候选原因。

## 修复门禁

修复必须最小化：

- 只改已证明根因对应的代码。
- 不引入无关重构。
- 不添加没有调用链证据支持的防御式判断。
- 不为了让测试通过而改变需求语义。
- 不删除或绕过关键业务逻辑。
- 不用临时日志、sleep、重试或兜底逻辑掩盖根因。

如果修复需要改变已确认计划外的逻辑，先说明原因、影响范围和替代方案，并等待用户确认。

## 回归门禁

修复后必须重新运行同一条复现流程：

- 修复前失败的入口，修复后仍走同一入口。
- 修复前观察的状态或副作用，修复后用同一方式检查。
- 如果新增了回归测试，该测试应先能在旧逻辑上失败，再能在新逻辑上通过；无法证明时必须说明原因。

不能用更低层、更短路径或只测 helper 的测试替代同路径回归。

## 汇报要求

最终汇报必须包含：

- 复现入口和复现命令。
- 修复前失败证据。
- 根因和关键代码位置。
- 修改摘要。
- 修复后同路径验证结果。
- 仍未验证的范围和风险。

缺少复现证据、根因证据或同路径验证时，只能说“已调查到当前状态”，不能说“已修复”。
