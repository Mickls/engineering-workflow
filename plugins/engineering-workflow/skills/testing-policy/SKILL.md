# 测试策略

## 概述

测试要证明用户关心的行为，而不是证明某个内部函数刚好返回某个中间值。优先从完整调用链路验证目标逻辑。

一个有证明力的链路测试，优先于多个低价值的内部实现测试。不要用测试数量制造覆盖感。

如果正在诊断 bug，先按 `diagnosis-workflow` 或 `incident-debugging` 建立反馈循环，再决定测试落点。

## 判断是否需要测试

以下情况应新增或更新测试：

- 行为变化。
- bugfix。
- 重要边界条件。
- public API/proto/schema 变更。
- 数据库 query 或 migration 行为。
- 权限、租户、并发、错误处理、重试行为。
- 调试过程中发现的回归场景。

以下情况可以跳过新增测试：

- 纯格式化。
- 注释。
- 简单文案。
- 无行为变化的重命名。
- 现有测试已经覆盖的机械结构整理。

跳过测试时必须说明：

- 为什么该修改没有行为影响。
- 检查了哪条调用链或 public contract。
- 现有相关测试是否已经覆盖该区域。
- 仍然运行了哪些已有检查，或为什么没有有意义的检查可运行。

## 测试范围

优先从相关调用链的顶层入口进入测试：

- `HTTP/RPC handler`、route、controller、resolver。
- `event/job` 接收处。
- `CLI` 命令。
- 页面或用户操作入口。
- public package/module API。

只有满足以下条件时，才使用更低层的 service/package/module 入口：

- 它本身就是该行为的稳定 public contract。
- handler、route、页面、job 或外层没有相关的参数绑定、校验、权限、租户、事务、副作用或错误映射逻辑。
- 已有测试已经覆盖被省略的外层。
- 明确说明为什么低层测试足以证明目标行为。

如果真实需求只能通过更高层 workflow 观察，不要只测试被改动的 helper 函数。

如果用户明确要求测试完整链路，必须从真实入口或等价顶层入口进入；不得降级为直接调用某个内部函数，除非先说明无法使用顶层入口的具体原因并获得用户确认。

## 测试价值门禁

新增或修改测试前，必须先明确这个测试证明什么用户可见行为或稳定 contract。

如果存在 `.codex/engineering-workflow/issues/` 下的 `design.md` / `plan.md` 关键约束覆盖表，测试设计必须从这些约束出发，而不是从“改了哪个函数”出发。非轻量 no-doc 例外也必须从最小关键约束清单出发。

高价值测试通常满足：

- 从真实入口或稳定 public contract 进入。
- 能观察用户关心的返回值、状态变化、数据库副作用、消息副作用、文件副作用或错误映射。
- 覆盖这次需求或 bugfix 的关键风险。
- 失败时能说明真实行为不符合预期，而不是只说明内部实现换了写法。

低价值测试应避免：

- 同一条路径已经被链路测试覆盖，又重复测试路径中每个 helper 的中间返回值。
- 只断言 mock 被调用、内部函数被调用、局部变量形状或偶然排序。
- 为了覆盖率把同一行为拆成多个没有独立风险的测试。
- 测试期望复制生产逻辑，导致生产和测试一起错。
- mock 掉要验证的核心逻辑，然后断言 mock 输出。

如果一个完整链路测试已经能证明目标行为，不要再为同一路径补一串低层测试。只有存在独立复杂逻辑、难以从链路触达的分支、稳定 public contract 或重要边界条件时，才补充低层测试，并说明它补充证明了什么。

单元测试仍适合：

- 复杂纯逻辑。
- 难以通过链路测试触达的分支。
- 边界条件。
- 小算法的确定性验证。
- 稳定 public contract 的直接验证。

## 测试覆盖矩阵门禁

非轻量需求需要维护测试覆盖矩阵，可写在 `plan.md`、对应 `.codex/engineering-workflow/issues/` 记录或 no-doc 临时测试计划段落中。

推荐格式：

```md
| 约束 ID | 测试/验证入口 | 测试名或命令 | 关键断言 | 覆盖状态 |
| --- | --- | --- | --- | --- |
| C-001 | HTTP/RPC/service/integration | TestXxx | 检查真实返回/状态/副作用 | planned |
```

规则：

- 每个关键约束 ID 都必须有测试、手动验证、代码审查证据，或明确不可自动验证原因；no-doc 例外不能因为没有完整 design/plan 而跳过核销。
- 一个高价值链路测试可以覆盖多个约束 ID；不要为同一条路径的内部层级重复写测试。
- 高风险约束必须有可观察断言，例如数据库状态、外部副作用、返回错误、状态机变化、任务入队、文件变化。
- `不能发生` 类约束必须有反向断言，例如“不删除”“不重复写入”“不自动处理”“不绕过权限”。
- 对 existing / non-existing、手动 / 自动、root 内 / root 外、include / exclude、同步 / 异步等分叉语义，不能只测其中一个分支后默认覆盖全部。
- 如果某个约束只能手测，必须写清手测入口、步骤和观察点；不能用“代码看起来对”替代。
- 测试运行后回填真实测试名、命令和结果；不能只写“相关测试通过”。

## 测试设计

编码前先明确：

- 接收的输入。
- 期望输出。
- 副作用。
- 期望错误行为。
- 受影响调用链。
- 已有 fixture 和 mock。
- 测试入口选择：为什么从这个入口测试。
- 证明目标：这个测试失败时能证明哪个用户行为或稳定 contract 出错。

覆盖范围应包括：

- 主要成功路径。
- 关键失败路径。
- 当前变更的回归路径。
- 重要边界条件。

不要机械要求每一项都拆成独立测试。优先用少量高价值测试覆盖完整路径；只有不同风险需要不同输入或不同入口时，才拆多个测试。

避免：

- 过度 mock 被测实现。
- 断言偶然的中间值。
- 在测试期望中复制生产逻辑。
- 让测试依赖 map 遍历顺序、sleep timing 或无关日志。
- 为同一条调用链重复测试多个内部层级。
- 未说明原因就把完整链路测试降级成内部函数测试。

mock 和 fixture 规则：

- 在外部依赖边界使用 fake，不要 mock 掉要验证的核心逻辑。
- 优先使用项目已有 fixture 和 builder。
- fixture 数据保持最小但真实。
- 不要把测试要证明的行为 mock 掉。
- 对数据库、队列、缓存、文件等副作用敏感的 bug，优先使用真实测试实例或项目已有 fake，确保能观察真实状态变化。

## bugfix 和回归测试

bugfix 测试必须先证明旧行为会失败，再证明修复后通过。不能在没有复现失败的情况下直接写“修复后测试”。

回归测试要求：

- 入口与用户问题一致或等价。
- 输入与线上触发条件一致或能解释为最小等价数据。
- 断言用户可见结果和关键副作用。
- 对数据误删、重复写入、幂等性、并发、重试类问题，必须断言持久化状态或可观察副作用。

如果 bug 是“连续两次调用同一个接口导致数据误删除”，测试必须连续两次调用同一个接口或等价顶层入口，并检查第二次调用后的真实数据状态。不得只调用内部删除函数或 service helper。

如果旧行为无法在本地测试中复现，必须说明已尝试的入口、命令、fixture、缺失条件和下一步需要的信息，不能靠猜测补一个不失败的测试。

## 测试失败处理

测试失败后，不要立刻根据失败表象修改生产逻辑。

先调查：

- 最近生产代码或测试发生了什么变化。
- 重构是否改变了调用链。
- 测试是否依赖了不再属于目标契约的内部行为。
- 失败断言证明的是用户可见行为，还是内部实现细节。
- 失败是否能从调用链顶层复现。
- 失败测试是否具有证明力，还是只是内部实现细节断言。

从顶层追踪相关代码：

- 入口。
- 输入构造。
- 依赖。
- 核心逻辑。
- 返回值。
- 副作用。
- 错误处理。

然后分类失败：

### 真实逻辑错误

只有当失败测试暴露目标调用链的真实行为错误时，才归类为逻辑错误。

修改生产逻辑前：

- 说明根因。
- 描述受影响行为和范围。
- 提出生产代码修改方案。
- 说明要运行的验证。
- 如果该逻辑修改不在已确认 plan 内，先获得用户允许。

### 测试需要调整

以下情况优先调整测试：

- 只覆盖内部实现细节。
- 过度 mock，已经不能代表真实调用链。
- 断言非关键中间值。
- 无法从调用链顶层证明预期行为。
- 在已确认需求变化后仍编码旧行为。
- 多个测试重复覆盖同一路径的不同内部点，没有独立证明价值。

### 环境或不稳定测试

单独调查环境问题：

- timing 和 race。
- 外部服务依赖。
- 顺序依赖。
- 脏本地状态。
- 缺失 fixture。

优先使用确定性等待和明确 fixture，不要直接增加 timeout。

## 失败报告模板

测试失败影响下一步行动时，报告：

- 复现命令。
- 失败测试名和断言。
- 被测入口或调用链层级。
- 失败分类：生产逻辑、测试不匹配、环境/不稳定、未知。
- 支撑该分类的证据。
- 下一步：调整测试、请求允许修改生产逻辑，或继续调查。

## 测试结果汇报

汇报测试结果时包含：

- 运行的命令。
- 结果。
- 失败测试，如有。
- 失败属于生产逻辑、测试不匹配还是环境问题。
- 被测入口层级：真实入口、public contract、service/package、内部 helper。
- 为什么该入口足以证明目标行为。
- 剩余风险。
