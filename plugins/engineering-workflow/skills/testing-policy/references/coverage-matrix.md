# 测试覆盖矩阵

非轻量需求需要维护测试覆盖矩阵，可写在 `plan.md`、对应 `.codex/engineering-workflow/issues/` 记录或 no-doc 临时测试计划中。

推荐格式：

```md
| 约束 ID | 测试/验证入口 | 测试名或命令 | 关键断言 | 覆盖状态 |
| --- | --- | --- | --- | --- |
| C-001 | HTTP/RPC/service/integration | TestXxx | 检查真实返回/状态/副作用 | planned |
```

规则：

- 每个关键约束 ID 都必须有测试、手动验证、代码审查证据，或明确不可自动验证原因。
- no-doc 例外不能因为没有完整 design/plan 而跳过核销。
- 一个高价值链路测试可以覆盖多个约束 ID；不要为同一条路径的内部层级重复写测试。
- 高风险约束必须有可观察断言，例如数据库状态、外部副作用、返回错误、状态机变化、任务入队、文件变化。
- `不能发生` 类约束必须有反向断言，例如“不删除”“不重复写入”“不自动处理”“不绕过权限”。
- 对 existing / non-existing、手动 / 自动、root 内 / root 外、include / exclude、同步 / 异步等分叉语义，不能只测其中一个分支后默认覆盖全部。
- 如果某个约束只能手测，必须写清手测入口、步骤和观察点；不能用“代码看起来对”替代。
- 测试运行后回填真实测试名、命令和结果；不能只写“相关测试通过”。
