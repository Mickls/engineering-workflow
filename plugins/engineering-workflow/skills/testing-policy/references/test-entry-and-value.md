# 测试入口和测试价值

## 入口选择

优先从相关调用链的顶层入口进入测试：

- `HTTP/RPC handler`、route、controller、resolver。
- `event/job` 接收处。
- `CLI` 命令。
- 页面或用户操作入口。
- public package/module API。

只有满足以下条件时，才使用更低层的 service/package/module 入口：

- 它本身就是该行为的稳定 public contract。
- handler、route、页面、job 或外层没有相关参数绑定、校验、权限、租户、事务、副作用或错误映射逻辑。
- 已有测试已经覆盖被省略的外层。
- 明确说明为什么低层测试足以证明目标行为。

不要只测试被改动的 helper 函数来替代真实 workflow。

## 测试价值

新增或修改测试前，先明确这个测试证明什么用户可见行为或稳定 contract。

高价值测试通常满足：

- 从真实入口或稳定 public contract 进入。
- 能观察用户关心的返回值、状态变化、数据库副作用、消息副作用、文件副作用或错误映射。
- 覆盖这次需求或 bugfix 的关键风险。
- 失败时能说明真实行为不符合预期，而不是只说明内部实现换了写法。

低价值测试应避免：

- 同一路径已被链路测试覆盖，又重复测试路径中每个 helper 的中间返回值。
- 只断言 mock 被调用、内部函数被调用、局部变量形状或偶然排序。
- 为了覆盖率把同一行为拆成多个没有独立风险的测试。
- 测试期望复制生产逻辑，导致生产和测试一起错。
- mock 掉要验证的核心逻辑，然后断言 mock 输出。

单元测试仍适合复杂纯逻辑、难以通过链路测试触达的分支、边界条件、小算法确定性验证和稳定 public contract 的直接验证。

## 测试设计

编码前先明确输入、期望输出、副作用、期望错误行为、受影响调用链、已有 fixture/mock、测试入口选择和证明目标。

覆盖主要成功路径、关键失败路径、当前变更的回归路径和重要边界条件。不要机械要求每一项都拆成独立测试，优先用少量高价值测试覆盖完整路径。

mock 和 fixture：

- 在外部依赖边界使用 fake，不要 mock 掉要验证的核心逻辑。
- 优先使用项目已有 fixture 和 builder。
- fixture 数据保持最小但真实。
- 对数据库、队列、缓存、文件等副作用敏感的 bug，优先使用真实测试实例或项目已有 fake，确保能观察真实状态变化。
