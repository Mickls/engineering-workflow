# 测试入口和测试价值

## 编码前验收场景

非轻量行为切片编码前必须绑定已批准的验收场景，至少记录：

- 场景 ID 和覆盖的约束 ID。
- 真实入口或稳定 public contract。
- 前置数据、权限或系统状态。
- 用户或调用方执行的操作。
- 用户可见返回、页面、文件或状态结果。
- 数据库、消息、任务、文件等关键副作用。
- 与当前风险相关的失败行为。

测试从验收场景推导，不能先实现再根据代码现状定义“正确结果”。实现中发现验收场景无法成立时，先重新澄清或修改已批准 design/plan，不能让测试和实现一起偏离用户目标。

## 入口选择

优先从相关调用链的顶层入口进入测试：

- `HTTP/RPC handler`、route、controller、resolver。
- `event/job` 接收处。
- `CLI` 命令。
- 页面或用户操作入口。
- public package/module API。

只有满足以下条件时，才使用更低层的 service/package/module 入口：

- 它本身就是该行为的稳定 public contract。
- handler、route、页面、job 或外层没有相关参数绑定、校验、权限、租户、事务、副作用或错误映射逻辑。
- 已有测试已经覆盖被省略的外层。
- 明确说明为什么低层测试足以证明目标行为。

不要只测试被改动的 helper 函数来替代真实 workflow。

## 测试价值

新增或修改测试前，先明确这个测试证明什么用户可见行为或稳定 contract。

高价值测试通常满足：

- 从真实入口或稳定 public contract 进入。
- 能观察用户关心的返回值、状态变化、数据库副作用、消息副作用、文件副作用或错误映射。
- 覆盖这次需求或 bugfix 的关键风险。
- 失败时能说明真实行为不符合预期，而不是只说明内部实现换了写法。

低价值测试应避免：

- 同一路径已被链路测试覆盖，又重复测试路径中每个 helper 的中间返回值。
- 只断言 mock 被调用、内部函数被调用、局部变量形状或偶然排序。
- 为了覆盖率把同一行为拆成多个没有独立风险的测试。
- 测试期望复制生产逻辑，导致生产和测试一起错。
- mock 掉要验证的核心逻辑，然后断言 mock 输出。

单元测试仍适合复杂纯逻辑、难以通过链路测试触达的分支、边界条件、小算法确定性验证和稳定 public contract 的直接验证。

## 测试设计

编码前从已批准场景明确输入、期望输出、副作用、期望错误行为、受影响调用链、已有 fixture/mock、测试入口选择和证明目标。

覆盖主要成功路径、关键失败路径、当前变更的回归路径和重要边界条件。不要机械要求每一项都拆成独立测试，优先用少量高价值测试覆盖完整路径。

mock 和 fixture：

- 在外部依赖边界使用 fake，不要 mock 掉要验证的核心逻辑。
- 优先使用项目已有 fixture 和 builder。
- fixture 数据保持最小但真实。
- 对数据库、队列、缓存、文件等副作用敏感的 bug，优先使用真实测试实例或项目已有 fake，确保能观察真实状态变化。

## 资源、隔离和重跑预算

- plan 应记录高成本 integration/build 的缓存目录、并行度、临时目录、fixture 隔离和清理边界；不得默认把大缓存写入用户系统盘。
- 优先复用同一 source diff 和同一验收场景下刚完成的有效结果。只有相关生产/测试代码、fixture、配置或依赖变化时才重跑高成本验证，不因每个微型 review 单元机械重跑全套。
- 集成测试必须只断言和清理当前测试创建的数据；历史残留或共享环境噪声不得通过扩大清理范围解决，也不得被误判为生产逻辑失败。
- codegen、migration、wire 和需要用户操作的测试环境准备集中到 plan 声明的自然里程碑，减少反复人工接力。
- 测试或构建异常增长磁盘、内存或运行时间时，停止新增并发任务，确认实际缓存位置和资源占用后再继续；交付时报告资源边界和未执行项。
