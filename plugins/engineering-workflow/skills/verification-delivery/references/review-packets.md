# 行为 / 风险 Review 包

本文件是 HITL 停点和最终交付 review 包的唯一 owner。目标是让用户先理解为什么改、实际行为和风险，不要求先阅读完整 diff 或技术术语。

## 用户 Review 区

默认先提供以下内容：

1. **用户能力**：本轮新增、改变或修复了什么可观察行为。
2. **目标关系**：为什么这部分工作属于已批准目标和当前切片。
3. **方案偏离**：是否偏离 design/plan；没有则明确写“无”。
4. **行为证据**：对应验收场景、真实入口、运行结果和关键副作用。
5. **剩余风险**：未验证环境、manual-only 步骤、可接受影响和未知。
6. **需要用户 review**：最多三个必须由用户处理或判断的事项；没有则写“无”。

用户 Review 区使用普通语言和实际场景。不得只写“测试通过”“已按方案实现”，也不得要求用户先阅读完整 diff 才能理解改动目的。

## 技术证据区

用户 Review 区之后再列：

- 关键文件和调用链。
- 约束 ID 到实现落点的映射。
- 实际运行的 lint/test/build 命令、退出状态和失败分类。
- 代码风格依据、相邻实现或项目契约。
- diff 统计、review budget 口径和排除项。
- 防御式代码、依赖、抽象、错误、日志和架构路径审查结果。

技术证据用于复核，不把用户不理解的术语包装成新的确认问题。发现需要用户决定时，返回 `requirements-workflow` 的问题准入和知情确认流程。

## HITL 停点

HITL review 包还必须说明：

- 实际触发场景和证据。
- 命中的 stop trigger。
- 受影响约束和后续切片。
- 为什么 agent 不能按项目证据自行继续。

只请求用户判断行为、风险或外部动作；普通技术实现继续由 agent 按项目模式负责。
